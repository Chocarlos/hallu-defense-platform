# Hallu Defense v0.1.0 release plan

## Status

`blocked` — this document defines the first public release contract but does not claim that a release exists or that the current branch is accepted.

The product maturity for `v0.1.0` is **technical alpha**. The version establishes a reproducible public baseline; it does not certify any specific deployment as production-ready.

## Release subject

The release must be created from one exact annotated, signed `v0.1.0` tag. The tag object, peeled source commit, protected control commit, workflow identity, and artifact digests must agree with the evidence produced by `.github/workflows/release.yml`.

Do not create or move the tag until the external controls in `docs/governance/repository-controls.md` are verified.

## Distribution contract

The initial release is an evidence-bearing bundle, not a promise that every workspace is published to a package registry.

Expected subjects from the existing release workflow:

- source archive for the exact signed commit;
- reproducible Python wheel for the API package;
- Python and Node runtime lock snapshots;
- CycloneDX SBOMs for the governed runtime dependency sets;
- Docker archives for the ten first-party images built by the workflow;
- per-image scan reports and image/source bindings;
- checksums, source statement, artifact-envelope bindings, and GitHub attestations.

The current workflow does **not** push images to a container registry and does **not** publish npm packages. Registry manifest digests and package-registry provenance therefore remain outside this release contract.

## Supported installation paths

For `v0.1.0`, the documented paths are:

1. clone or extract the exact source release and run the repository quick start;
2. use Docker Compose for the local integration profile;
3. consume the verified Python wheel where the API-only package is sufficient;
4. inspect or load a release image archive only after independently verifying its checksum and attestation.

The SDK, contracts, MCP server, agent adapters, and console remain monorepo workspaces until a separate package publication process is implemented and accepted.

## Required acceptance evidence

Before the tag is created, record for the exact candidate commit:

- successful `ci`, `evals`, and `security` workflow runs;
- proportional live workflow results, with every skipped external lane named explicitly;
- repository diff review and `git diff --check` evidence;
- license, provenance, changelog, security policy, and contribution-governance review;
- protected default branch, protected `v*` tags, release Environment, and private vulnerability reporting;
- remaining known defects, including any unresolved ingestion crash/restart behavior;
- confirmation that no secret, credential, private key, tenant data, or private recovery artifact is included.

## Release procedure

1. Select and freeze one candidate commit.
2. Run all required pull-request and live evidence against that commit.
3. Update the QA ledger without converting environment-specific evidence into a global claim.
4. Merge only through the protected default-branch process.
5. Create the annotated, signed `v0.1.0` tag from the accepted commit.
6. Dispatch the release workflow from the protected default branch.
7. Verify every build, scan, checksum, SBOM, source binding, and attestation independently.
8. Create GitHub release notes from `CHANGELOG.md`, linking the exact evidence and limitations.
9. Publish artifacts only through a separately authorized operation; publication must not change their bytes.

## Explicit non-goals

`v0.1.0` does not claim:

- universal hallucination elimination;
- production certification for an arbitrary environment;
- compatibility with providers beyond the documented matrix;
- managed-service failover, scale, restore, or external identity-provider acceptance;
- registry publication of images or npm workspaces;
- activation of the public demo intake.
